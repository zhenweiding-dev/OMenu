# OMenu 项目分析报告

## 📋 产品理念总结

### 核心问题
OMenu 应对的是现代家庭的真实痛点：**餐饭规划碎片化**。当前用户面临的工作流程：
1. 周末讨论下周菜单（低效、容易遗忘）
2. 手动制作购物清单（容易遗漏、重复）
3. 每日临时决定烹饪内容和步骤（浪费时间、不科学）

### 产品定位
一个 **AI 驱动的全流程餐饭规划系统**，将分散的决策和执行整合成一个连贯的、智能的、可执行的系统。

### 关键价值主张
- **时间节省**：从讨论→规划→购物→烹饭的整个过程自动化
- **决策支持**：AI 基于多维度约束（预算、时间、营养、偏好）生成合理方案
- **执行指导**：每日精确的时间轴和步骤，消除"现在做什么"的疑虑
- **个性化**：考虑家庭成员偏好、饮食限制、烹饪能力

## 🔧 实现思路分析

### 整体架构流程

```
用户需求输入 → AI 生成方案 → 用户审查调整 → 购物清单生成 → 每日执行指导
```

### 核心模块

#### 1. **需求收集模块**（多维度约束）
- **饮食维度**：特殊食物、饮食类型、限制条件
- **时间维度**：规划周期、单餐烹饪时间、可用烹饪窗口
- **资源维度**：人数、预算、烹饪能力
- **执行方式**：自动生成 vs 半手动选择

**评价**：覆盖面全，但问卷可能过长。建议分阶段引导，简化初次使用。

#### 2. **AI 生成模块**（Gemini 3）
核心职责：
- 在复杂约束下优化菜单组合
- 考虑营养平衡和食材复用
- 估算准备和烹饪时间

**技术思考**：
- 需要一个高质量菜谱数据库（格式统一、信息完整）
- Gemini 的强项是推理和多模态，完全可以胜任这个任务
- 可考虑两步策略：快速生成初稿 + 详细信息按需加载（节省 token）

#### 3. **审查调整模块**
- 日历视图展示整周计划
- 支持菜肴交换、删除、添加
- 实时购物清单更新

**评价**：UX 设计关键点。建议用拖拽交换、一键替换等交互简化调整流程。

#### 4. **购物清单模块**
- 自动聚合和数量计算
- 按类别分组（生鲜、肉类、调料等）
- 支持调整份数后自动重算

**评价**：相对直接，但要考虑：
- 食材标准化（如"鸡肉 500g" vs "鸡胸肉一块"）
- 库存管理（用户可能已有某些食材）

#### 5. **每日执行模块**
- 精确时间轴（10:00am 准备食材 15 分钟）
- 实时状态追踪（已完成→灰显）
- 可选通知和振动提醒

**评价**：这是 **差异化亮点**。很多食谱应用有菜谱，但精确到分钟的时间管理罕见。

## ✨ 核心优势

### 1. **真实的用户需求**
- 不是伪需求，而是数百万家庭的日常困扰
- 用户已经在用纸笔、Excel、群聊等不便方案
- 有明确的支付意愿（时间节省的价值）

### 2. **AI 优势发挥充分**
- Gemini 3 的多模态能力可扩展应用：
  - 语音输入偏好（"今周想吃点清淡的"）
  - 拍照识别冰箱食材（未来功能，高价值）
  - 图像识别烹饪步骤（辅助参考）
- 推理能力用于复杂的约束优化（营养平衡、预算、时间、偏好权衡）

### 3. **完整的价值链闭环**
从"想吃什么" → "买什么" → "怎么做" 的端到端覆盖，而非单点工具。

### 4. **多维度的个性化**
不止推荐菜肴，而是推理每个家庭的独特约束，生成定制方案。

## ⚠️ 潜在挑战

### 1. **菜谱数据库**
- **问题**：质量、覆盖度、格式统一性
- **风险**：数据不完整导致 AI 生成错误方案（如时间估计不准）
- **建议**：
  - 考虑与现有菜谱平台合作（小红书、下厨房等）
  - 或构建众包模式让用户补充菜谱
  - 初期可从高质量开源菜谱库开始

### 2. **时间估计的准确性**
- **问题**："炖汤 120 分钟" vs "炖汤 150 分钟"，取决于火力、食材大小、个人习惯
- **风险**：时间表不准会降低用户信任
- **建议**：
  - 提供时间范围而非点值
  - 允许用户基于历史调整
  - 学习用户的烹饪速度（ML 优化）

### 3. **初期冷启动问题**
- **问题**：AI 需要大量的用户行为数据才能精准推荐
- **风险**：前 100 个用户的体验可能较差
- **建议**：
  - 提供 "智能推荐" 和 "人工模板" 两种模式
  - 邀请美食博主/营养师设计初期模板
  - 实时收集用户反馈优化

### 4. **市场竞争**
- **竞争对手**：Mealime、Goodful、植物性饮食应用等
- **差异化关键**：精确到分钟的时间管理 + 多模态 AI（拍冰箱等）
- **建议**：聚焦在时间管理的精准性和执行的便利性上

### 5. **用户心理障碍**
- **问题**：用户可能不信任 AI 推荐，或觉得限制了选择自由
- **建议**：
  - 强调 "AI 辅助，用户决策" 的定位
  - 提供充分的调整和推翻选项
  - 展示 AI 的推理逻辑（"为什么推荐这个菜"）

### 6. **技术复杂度**
- **问题**：涉及 NLP、推理、图像识别、实时同步等
- **建议**：
  - 分阶段上线（MVP 先不含拍照识别）
  - 充分利用 Gemini API，减少自研
  - 后端架构要考虑可扩展性

## 💡 建议和优化方向

### 短期（MVP 阶段，0-3 个月）
1. **菜谱库**：优先集成 1000+ 高质量菜谱，覆盖常见菜系
2. **核心流程**：实现需求收集 → AI 生成 → 购物清单
3. **时间估计**：使用保守估计 + 用户反馈循环
4. **用户研究**：邀请 10-20 个家庭进行 Beta 测试，收集反馈

### 中期（1-6 个月）
1. **审查调整 UI**：优化拖拽、交换等交互，降低调整成本
2. **每日执行优化**：测试通知、振动、语音播报等执行模式
3. **学习引擎**：基于用户历史改进时间估计和推荐
4. **社区功能**：允许用户分享成功的菜单组合

### 长期（6-12 个月）
1. **多模态输入**：拍冰箱识别食材，基于现有食材优化建议
2. **营养分析**：详细的营养成分展示和平衡提示
3. **采购优化**：集成超市价格，提供采购路线优化
4. **家庭协作**：多用户实时协作规划和采购清单

### 商业模式建议
- **免费版**：基础菜单规划、购物清单
- **会员版**（$4.99/月或 $49.99/年）：高级菜系库、营养分析、多人协作、采购优化
- **B2B**：向食材配送平台或食品公司授权 API
